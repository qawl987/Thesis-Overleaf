\section{Background and Motivation}
\label{sec:background}

\subsection{5G Slice Management}

3GPP describes a network slice instance as an end-to-end logical network composed of slice subnets across domains such as the core network, RAN, and transport network~\cite{3gpp_ts28530}. Management systems commonly decompose this responsibility into a network slice management function (NSMF) and domain-specific network slice subnet management functions (NSSMFs)~\cite{3gpp_ts28533}. This decomposition is useful, but it leaves the operator with the problem of mapping service objectives into domain-specific parameters.

For the core network, free5GC implements a 5G core following the service-based architecture of 3GPP Release 15~\cite{free5gc,3gpp_ts23501}. Relevant slice and QoS controls include S-NSSAI, subscriber data, session management, QoS flow identifiers, and 5QI. For the RAN, OCUDU exposes practical controls for slice-aware scheduling through S-NSSAI-specific parameters, including PRB policy ratios and priority~\cite{ocudu,ocudu_oran_gnb_doc}. The challenge is not the absence of knobs, but the lack of a unified layer that derives them from an operator intent.

\subsection{Kubernetes, Operators, and Nephio}

Kubernetes offers a declarative API and a reconciliation model that are well matched to intent management. Custom Resource Definitions (CRDs) extend the API with domain-specific objects~\cite{kubernetes_custom_resources}. Operators then reconcile desired state into actual runtime state~\cite{kubernetes_operator_pattern,kubernetes_controller}. This makes Kubernetes a natural substrate for expressing both high-level intent and low-level network-function configuration.

Nephio extends this model with package orchestration for telecom workloads. It uses Kubernetes Resource Model (KRM) packages, Porch package revisions, and GitOps synchronization to propagate configuration from a management cluster to workload clusters~\cite{nephio,google_config_sync,cncf_gitops_2025}. This approach provides version control, review, and rollback semantics for network configuration. It is especially attractive for RAN configuration because changes to cell-level scheduling policy can be represented as structured KRM resources before being rendered into the final OCUDU configuration.

\subsection{Gap}

Prior work on intent-based and autonomous network management proposes architectures for translating service objectives into network behavior~\cite{chirivella2023e2e,martini2023intent,tran2024ina,antonakoglou2025camino,nahum2025intent}. Recent work also studies GitOps performance for network automation~\cite{ghosh2025performance}. The remaining gap is a concrete end-to-end prototype that connects intent submission, cross-domain translation, Nephio-based RAN mutation, free5GC QoS configuration, and experimental validation with open-source 5G components. This paper addresses that gap.

Table~\ref{tab:positioning} highlights the design premise. Each component already solves part of the deployment problem, but none of them alone provides a complete path from service intent to realized RAN and core configuration. The proposed orchestrator is therefore intentionally small: it does not replace free5GC, OCUDU, Kubernetes, or Nephio. Instead, it connects their control surfaces and records the translation decisions needed to make slice fulfillment auditable.
