\chapter{System Model}

\section{系統架構設計}

\subsection{系統分層設計}
本研究設計並實作一套意圖驅動切片編排系統（Intent-Driven Slicing Orchestration System），旨在透過宣告式介面簡化網路切片之管理門檻。如圖 \ref{fig:system_arch} 所示，本系統在視覺架構上劃分為三個主要平面，並依據職能細分為四個管理層級，以符合 3GPP 服務化管理架構（SBMA）之解耦原則。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\linewidth]{img/3.1.1-arch.png}
    \caption{本系統之意圖驅動跨網域切片編排架構圖}
    \label{fig:system_arch}
\end{figure}

\begin{enumerate}
    \item \textbf{服務消費者層（Service Consumer Layer）}：對應架構圖中 Layer 1 之 Service Consumer Domain。此層級為管理需求之發起者，扮演 \textbf{MnS Consumer} 之角色。消費者透過 \textbf{Kubernetes API Server} 提供的 \textbf{Intent Management MnS Interface}，下發宣告式之 \texttt{E2EQoSIntent} 自定義資源。使用者僅需根據業務需求定義高階期望指標，例如頻寬期望（\texttt{bandwidth: "high"}）、延遲期望（\texttt{latency: "ultra-low"}）或資源共享策略（\texttt{resourceShare: "Full"}），無需介入底層網域之特定技術配置。

    \item \textbf{意圖編排與協調層（Intent Orchestration Layer）}：對應架構圖 Layer 2 之核心組件。本層級由 \textbf{E2E Orchestrator} 構成，實質承擔 \textbf{NSMF / Intent Handler} 之職能。其負責監聽北向介面之意圖請求，並執行意圖轉譯（Intent Translation），將抽象 SLA 期望精確映射為網域特定參數（如 Core 網域之 5QI 與 RAN 網域之 PRB 比例）。

    \item \textbf{供應與南向介面層（Southbound Interface Layer）}：同樣位於 Layer 2 之管理叢集（Management Cluster）內，負責執行跨網域之編排指令下發。針對 RAN 網域，本系統整合 \textbf{Nephio Porch} 與 \textbf{Kpt Pipeline}，透過 \textbf{Git Repo} 實施配置套件之變更與發布；針對 Core 網域，則透過封裝之 \textbf{REST Client} 執行混合式編排，直接與核心網管理介面進行通訊。

    \item \textbf{網域執行與實施層（Domain Execution Layer）}：對應架構圖 Layer 3 之 NSSMF Domain。此層級負責將編排結果落實於實體網路環境，包含負責無線存取網網域調和之 \textbf{srsran-operator} 與負責核心網網域調和之 \textbf{free5gc-operator}。各 Operator 透過 \textbf{ConfigSync} 監聽期望狀態，並驅動實體網元（如 srsRAN 之 CU/DU 與 free5GC 之各個 NF）執行資源隔離與 QoS 配置。
\end{enumerate}

\subsection{意圖驅動工作流：從 SLA 期望到實體佈署之參數演化}
\label{sec:intent_workflow_trace}

為具體展示本系統如何落實「意圖驅動（Intent-driven）」理念，本節以同時請求 eMBB 與 URLLC 雙切片的情境為例，追蹤意圖參數在系統各層級間的轉譯與流轉過程。在此工作流中，系統將高度抽象的服務層級協議（SLA）期望，逐步解耦並轉化為實體存取網（DU）與核心網的底層配置。

\subsubsection{階段一：意圖下發 (Intent Submission)}
流程始於意圖擁有者（MnS Consumer）下發宣告式資源 \texttt{E2EQoSIntent}。如清單 \ref{lst:input_intent} 所示，該意圖隱蔽了所有 5G 底層技術細節，僅宣告業務端對網路效能的純粹期望。例如，eMBB 業務請求 \texttt{high} 頻寬與部分資源共享（\texttt{Partial}），而 URLLC 業務則嚴格要求 \texttt{ultra-low} 延遲與完全資源隔離（\texttt{Full}）。

\begin{lstlisting}[language=yaml, caption={MnS Consumer 下發之 E2EQoSIntent 完整定義}, label={lst:input_intent}]
apiVersion: e2e.intent.domain/v1alpha1
kind: E2EQoSIntent
metadata:
  name: slices-intent
spec:
  intentGroups:
    - id: embb
      contexts:
        targetUEs: ["208930000000001"]
      expectations:
        sliceType: eMBB
        bandwidth: high
        resourceShare: Partial
    - id: urllc
      contexts:
        targetUEs: ["208930000000002"]
      expectations:
        sliceType: URLLC
        latency: ultra-low
        resourceShare: Full
\end{lstlisting}

\subsubsection{階段二：跨網域意圖轉譯 (Cross-Domain Translation)}
E2E Orchestrator 接收到上述意圖後，其內建的轉譯引擎會依據 3GPP 規範與本地資源池狀態，將 SLA 期望轉譯為具體的網路參數，並回寫至該 CRD 的 \texttt{status} 欄位中（見清單 \ref{lst:translated_params}）。在此階段：
\begin{itemize}
    \item \textbf{Core 參數}：URLLC 的超低延遲期望被精確映射至 \texttt{fiveQI: 85}。
    \item \textbf{RAN 參數}：eMBB 的 \texttt{Partial} 共享被映射為 PRB 比例 0-50\%（優先級 10）；而 URLLC 的 \texttt{Full} 隔離則被賦予最高 100\% 的 PRB 獲取權與極高的排程優先級（Priority 200）。此外，切片識別碼（SD）亦在此階段完成從十六進制（如 0x112233）至十進制（1122867）的系統內碼轉換。
\end{itemize}

\begin{lstlisting}[language=yaml, caption={E2E Orchestrator 轉譯後之參數狀態 (片段)}, label={lst:translated_params}]
status:
  intentGroupStatuses:
    # ... (eMBB 轉譯略) ...
    - id: urllc
      translatedParams:
        coreParams:
          fiveQI: 85
          qfi: 85
        ranParams:
          sst: 1
          sd: 1122867      # 對應 0x112233
          minPrbPolicyRatio: 0
          maxPrbPolicyRatio: 100
          priority: 200
\end{lstlisting}

\subsubsection{階段三：南向配置變更與封裝 (Declarative Package Mutation)}
完成轉譯後，編排器透過 Nephio Porch 對南向存取網域進行配置修改（Mutation）。編排器僅精準替換與切片相關之配置，如清單 \ref{lst:porch_mutation} 所示，\texttt{SrsRANCellConfig} 中的 \texttt{spec.slicing} 陣列被寫入前述的 PRB 與 Priority 參數，而其他實體層參數（如頻段、頻寬）則完整繼承自基礎藍圖（Blueprint）。

\begin{lstlisting}[language=yaml, caption={Porch 修改後之 SrsRANCellConfig 宣告 (片段)}, label={lst:porch_mutation}]
spec:
  # ... (實體層參數沿用 Blueprint) ...
  slicing:
    - sst: 1
      sd: 66051
      schedCfg:
        minPrbPolicyRatio: 0
        maxPrbPolicyRatio: 50
        priority: 10
    - sst: 1
      sd: 1122867
      schedCfg:
        minPrbPolicyRatio: 0
        maxPrbPolicyRatio: 100
        priority: 200
\end{lstlisting}

\subsubsection{階段四：領域實體化與組態生成 (Domain Implementation)}
當宣告式配置透過 GitOps 機制同步至工作負載叢集後，\texttt{srsran-operator} 將攔截此變更。Operator 會遞迴查找 \texttt{NFConfig} 內關聯的所有配置檔，並將 KRM 格式的抽象定義，最終轉化為實體 Distributed Unit (DU) 所需之應用層組態檔。如清單 \ref{lst:final_configmap} 所示，這標誌著從抽象意圖到實體 srsRAN \texttt{gnb-config.yml} 的最後一哩路，確保了切片排程演算法的正確掛載與執行。

\begin{lstlisting}[language=yaml, caption={srsRAN Operator 生成之最終 DU ConfigMap (片段)}, label={lst:final_configmap}]
data:
  gnb-config.yml: |
    # ... (ZMQ 與 F1 介面配置) ...
    cell_cfg:
      # ...
      slicing:
        - sst: 1
          sd: 66051
          sched_cfg:
            min_prb_policy_ratio: 0
            max_prb_policy_ratio: 50
            priority: 10
        - sst: 1
          sd: 1122867
          sched_cfg:
            min_prb_policy_ratio: 0
            max_prb_policy_ratio: 100
            priority: 200
\end{lstlisting}

\section{意圖管理服務介面與端到端編排器實作}
\label{sec:e2e_orchestrator}

本節將深入探討端到端編排器（E2E Orchestrator）的內部架構與實作細節。作為系統中的 \textit{NSMF} 與 \textit{Intent Handler}，該編排器負責介接北向的意圖管理服務，並透過異質的南向介面協調核心網與存取網資源。本節首先定義基於 KRM 的宣告式意圖模型，接著解析跨網域意圖轉譯引擎之演算法，最後分別論述 RAN 與 Core 網域之編排路徑設計，並說明基於狀態回饋的閉環監控機制。

\subsection{宣告式意圖模型設計：\texttt{E2EQoSIntent} CRD 結構與 KRM 規範}
\label{subsec:intent_crd}

為落實 3GPP 所倡導的意圖驅動網路管理（Intent-driven Network Management），本研究開發了名為 \texttt{E2EQoSIntent} 的自定義資源定義（Custom Resource Definition, CRD），作為系統北向的意圖管理服務介面（Intent Management MnS Interface）。

相較於傳統基於 HTTP POST 方法的命令式（Imperative）切片建立 API（例如：\texttt{POST /slice}），本研究採用 Kubernetes 資源模型（Kubernetes Resource Model, KRM）來設計意圖介面，主要基於以下兩項架構優勢：
\begin{enumerate}
    \item \textbf{最終一致性（Eventual Consistency）的保障}：在傳統命令式架構中，若 API 呼叫過程遭遇網路中斷或網元失效，系統狀態極易陷入不一致。透過 KRM 的宣告式（Declarative）設計，MnS Consumer 僅需定義「期望的最終狀態（Desired State）」。底層的 Kubernetes 控制迴圈（Control Loop）將持續不斷地比較「當前狀態（Current State）」與期望狀態，並自動執行調和（Reconciliation），從而在分散式環境中提供強健的容錯與自癒能力。
    \item \textbf{業務意圖與基礎設施的解耦}：\texttt{E2EQoSIntent} 的資料結構被設計為高度抽象的服務層級協議（SLA），意圖擁有者無需具備 5G 底層技術領域知識（如 5QI 映射、PRB 分配算法），即可透過直觀的語意要求網路切片服務。
\end{enumerate}

清單 \ref{lst:input_intent} 展示了 \texttt{E2EQoSIntent} CRD 的 YAML 結構範例。該模型支援單一請求內包含多組意圖（\texttt{intentGroups}），每組意圖透過 \texttt{contexts} 綁定目標用戶設備（\texttt{targetUEs}，以 IMSI 識別），並在 \texttt{expectations} 區塊中宣告高階 SLA 指標，例如切片類型（\texttt{sliceType}）、延遲要求（\texttt{latency}）、頻寬需求（\texttt{bandwidth}）以及資源共享策略（\texttt{resourceShare}）。


\subsection{跨網域意圖轉譯引擎：SLA 指標至 5QI 與 PRB Ratio 之對映演算法}
\label{subsec:intent_translation}

在確立了宣告式意圖模型後，\textit{E2E Orchestrator} 內建的轉譯引擎（Translation Engine）負責將 \texttt{E2EQoSIntent} 中抽象的 SLA 期望（Expectations）解耦，並轉化為底層核心網與存取網可理解且可執行的領域特定參數。此轉譯過程消除了業務端高階需求與底層網路配置間的語意鴻溝（Semantic Gap）。

轉譯引擎之對映演算法分為「核心網（Core）參數轉譯」與「存取網（RAN）參數轉譯」兩個平行維度：

\subsubsection{核心網參數轉譯：基於 3GPP 規範之 5QI 映射}
核心網的轉譯目標為 5G 服務品質識別碼（5G QoS Identifier, 5QI）與對應的 QoS Flow 參數。本演算法嚴格遵循 3GPP TS 23.501 標準中所定義的 QoS 特性進行映射：
\begin{itemize}
    \item \textbf{延遲敏感型業務映射 (URLLC)}：演算法將意圖中的 \texttt{latency} 指標映射至 3GPP 規範的延遲階級。例如，當業務需求宣告為超低延遲（\texttt{ultra-low}）時，引擎會將其精確映射至 \textbf{5QI 85}。在 3GPP 標準中，5QI 85 具備「延遲臨界（Delay Critical）」特性，專為保障嚴苛的錯誤率與延遲限制（Delay Bound）所設計。次之的 \texttt{low} 與 \texttt{medium} 延遲期望，則分別降級映射至 5QI 82 與 5QI 84。
    \item \textbf{頻寬需求型業務映射 (eMBB)}：針對 \texttt{bandwidth} 指標，演算法依據頻寬需求的嚴格程度，對應至 3GPP 中階級不同的 5QI。最高級別的 \texttt{dedicated-high} 映射至具備保證位元速率（GBR）特性的 5QI 4；而 \texttt{high} 與 \texttt{standard} 則分別映射至優先級遞減的非保證位元速率（Non-GBR）類別，即 5QI 6 與 5QI 9。
\end{itemize}

\subsubsection{存取網參數轉譯：S-NSSAI 與無線資源隔離策略}
在存取網端，轉譯重點在於切片識別與無線實體資源區塊（Physical Resource Block, PRB）的調度參數分配：
\begin{itemize}
    \item \textbf{資源隔離比例 (PRB Policy Ratio)}：意圖中的 \texttt{resourceShare} 屬性決定了無線資源的實體隔離程度。若宣告為 \texttt{Full}（如 URLLC），引擎將 \texttt{maxPrbPolicyRatio} 設為 100\%，賦予該切片獲取全局可用頻寬的權限；若宣告為 \texttt{Partial}（如 eMBB），則將 \texttt{maxPrbPolicyRatio} 限制為 50\%，藉此預防單一切片耗盡所有無線資源，落實防飢餓（Starvation Prevention）的隔離目標。
    \item \textbf{排程優先級 (Scheduling Priority)}：為確保在無線資源面臨競爭時，關鍵型業務能被優先排程，演算法為 URLLC 切片配置了極高的 MAC 層排程權重（\texttt{priority: 200}），而一般 eMBB 切片則配置為基礎權重（\texttt{priority: 10}）。
\end{itemize}

\subsection{RAN 子網域編排路徑：基於 Nephio Porch 之宣告式封裝工作流}
\label{subsec:ran_orchestration_path}

在本系統的雙路徑派發架構中，存取網（RAN）子網域採用純宣告式（Declarative）的編排路徑。此路徑為本研究之核心實作貢獻之一，其突破了傳統命令式腳本在分散式環境下容易遭遇的狀態碎片化問題。\textit{E2E Orchestrator} 將 Nephio Porch 作為供應管理服務框架（Provisioning MnS Framework），透過操控 KRM 套件的生命週期，達成切片配置的自動化下發。

\subsubsection{基於狀態機之封裝編排工作流 (Package Orchestration Workflow)}
為了將轉譯後的 PRB Ratio 與 Priority 等參數安全地套用至存取網，\textit{E2E Orchestrator} 內建了 \texttt{PorchClient}，並實作了一套完整的套件變更（Package Mutation）工作流。該工作流嚴格遵循 Porch 狀態機的移轉規範，涵蓋以下自動化步驟：

\begin{enumerate}
    \item \textbf{建立草稿與提取 (Copy \& Pull)}：編排器首先以基礎藍圖（Blueprint）為範本，透過 \texttt{copy} 指令複製出一個全新的套件修訂版（PackageRevision）草稿，並將其拉取（Pull）至記憶體中進行處理。
    \item \textbf{意圖參數注入 (Mutate)}：編排器解析套件內的 \texttt{srscellconfig.yaml} 等資源檔，針對 \texttt{spec.slicing} 陣列進行精準替換。同時，視意圖需求動態修改 \texttt{PLMNConfig}，注入對應的切片識別碼（SST/SD），而未涉及切片的實體層參數則原封不動予以保留。
    \item \textbf{版本推播與提議 (Push \& Propose)}：完成參數替換後，編排器將變更後的套件推播（Push）回 Porch，並將套件狀態推進至 \texttt{Propose}，代表該配置已準備好接受審查。
    \item \textbf{批准發佈 (Approve)}：最終，編排器觸發 \texttt{Approve} 操作，正式發佈該套件。此動作將觸發底層 CISM (Cluster Infrastructure \& Service Management) 代理程式（如 ConfigSync）的同步機制，將配置派發至目標工作負載叢集。
\end{enumerate}

\subsubsection{宣告式編排之學術與工程價值}
透過上述基於 Nephio Porch 的工作流實作，本系統在 RAN 網域的編排上達成了兩項關鍵的雲原生工程特性：
\begin{itemize}
    \item \textbf{具備可追溯性之版本控制 (Version Control)}：所有針對 RAN 切片參數的修改，皆被封裝為不可變的修訂版本（PackageRevision）。這使得網路配置具備了類似軟體程式碼的「單一事實來源（Single Source of Truth, SSOT）」，當意圖發生衝突或配置失效時，系統具備退回（Rollback）至前一穩定版本的基礎能力。
    \item \textbf{確保狀態一致性之原子性變更 (Atomic Mutation)}：在傳統 REST API 連續呼叫的模式下，若中途發生網路異常，極易導致配置僅部分生效的「髒狀態（Dirty State）」。本系統透過 KRM 套件封裝，將 \texttt{SrsRANCellConfig}（PRB 資源劃分）與 \texttt{PLMNConfig}（網路識別廣播）等具備高度相依性的資源打包。當 ConfigSync 進行同步時，這些配置會被視為一個完整的邏輯單元進行套用，確保了變更的「原子性」——意即配置要麼全部生效，要麼全部不生效，從根本上消除了分散式系統中部分配置失敗的隱患。
\end{itemize}

\subsection{核心網子網域編排路徑：基於 REST API 之混合式異質整合機制}
\label{subsec:core_orchestration_path}

在本系統的架構設計中，核心網（Core Network）子網域採用「混合式編排（Hybrid Orchestration）」路徑。此設計不同於存取網（RAN）所採用的純宣告式套件編排，其核心考量在於現有開源核心網實作的功能限制，以及 5G QoS 機制的實施邏輯。

\subsubsection{異質網域編排之實作困境與限制}
在存取網端，如 \textit{srsRAN} 等實作允許透過修改網元組態檔（如 \texttt{cell\_cfg.slicing}）來定義特定切片的資源優先級與 PRB 比例。然而，鑑於現有開源核心網實作（如 \textit{free5GC}），其網路功能（NF）如 SMF 或 UPF 並不具備可供動態調整切片優先級的組態參數。核心網對切片服務品質（QoS）的控制，實質上是透過用戶設備（UE）的註冊資訊與服務合約（Subscription Data）進行綁定。

意圖模型中所定義的 \textit{5QI} 參數，必須透過核心網的註冊機制注入 UE 上下文（UE Context），方能於 PDU 會話建立過程中將 QoS 需求傳遞至存取網，使 RAN 識別該 UE 所屬之切片特性。因此，\textit{E2E Orchestrator} 無法如同處理 RAN 網域般，單純透過 \textit{Nephio Porch} 修改 SMF/UPF 的宣告式組態來達成意圖。

\subsubsection{混合式編排架構之實施}
為了克服上述限制，本系統採用「混合式編排」架構，其核心設計邏輯在於：在維持編排器北向介面（Intent Management MnS Interface）宣告式特性的同時，透過封裝 \textit{REST Client} 確保對異質網域之相容性。

具體實作上，\textit{E2E Orchestrator} 內建了 \texttt{Free5GCClient} 模組，將轉譯後的 5QI、S-NSSAI 與目標 IMSI 封裝為核心網可理解之 REST 指令：
\begin{enumerate}
    \item \textbf{動態用戶註冊}：針對 \texttt{E2EQoSIntent} 中定義的每一組 \texttt{targetUEs}，編排器透過 \texttt{RegisterSubscriber()} 介面自動於核心網建立對應用戶。
    \item \textbf{QoS 參數注入}：若用戶已存在，編排器則調用 \texttt{UpdateSubscriberQoS()}，將意圖轉譯所得之 5QI、SST 與 SD 參數注入該用戶的會話管理訂閱資料（Session Management Subscription Data）中。
\end{enumerate}

\subsection{意圖達成狀態監控：基於 CRD Status 之狀態回饋迴圈}
\label{subsec:intent_monitoring}

意圖驅動編排的核心在於「閉環控制（Closed-loop Control）」邏輯，而有效的狀態回饋是落實閉環控制的前提。在本系統中，\textit{E2E Orchestrator} 利用 Kubernetes CRD 的 \texttt{status} 子資源實作了意圖監控機制，這不僅是用於呈現系統當前狀態，更構成了狀態回饋迴圈（Status Feedback Loop）的觀察基礎。

\subsubsection{意圖達成狀態定義 (Fulfillment State)}
為了使系統具備標準化的評估準則，本研究參考 3GPP TS 28.312 規範，在 \texttt{E2EQoSIntent} 的狀態模型中引入了「達成狀態（Fulfillment State）」指標。編排器會根據各子網域的回報，動態更新以下狀態：
\begin{itemize}
    \item \textbf{FULFILLED}：代表 RAN 與 Core 網域均已正確套用轉譯參數，且底層資源已成功實體化。
    \item \textbf{NOT\_FULFILLED}：代表意圖剛下發或轉譯、派發過程中遭遇錯誤（如 REST API 連線失敗或 Porch 套件衝突）。
    \item \textbf{DEGRADED}：當系統偵測到實體資源狀態（如 Pod 運行狀態）偏離預期，但配置尚未失效時標註。
\end{itemize}

\subsubsection{多維度狀態回饋結構}
如清單 \ref{lst:intent_status} 所示，系統的 \texttt{status} 欄位被設計為階層式結構，以確保監控的可追溯性：
\begin{enumerate}
    \item \textbf{處理階段 (Phase)}：追蹤編排器的處理進度（如 \texttt{Processing} 或 \texttt{Applied}）。
    \item \textbf{觀察世代 (Observed Generation)}：紀錄編排器最後一次處理的意圖版本，結合 Kubernetes 的 \texttt{metadata.generation} 機制，確保編排器能精確偵測到意圖的任何微小變更並觸發重新調和。
    \item \textbf{網域特定狀態 (Domain Status)}：細分核心網（Core）與存取網（RAN）的實施狀態。例如，核心網狀態會反映 \textit{free5GC} 用戶註冊是否成功；存取網狀態則反映 \textit{Nephio} 套件是否完成 Approve 流程。
    \item \textbf{達成目標 (Achieved Targets)}：將底層實施結果重新映射回業務語意（如 \texttt{bandwidth: achieved}），為 MnS Consumer 提供直觀的業務達成率報告。
\end{enumerate}

\begin{lstlisting}[language=yaml, caption={\texttt{E2EQoSIntent} 狀態回饋與監控資訊結構 (範例)}, label={lst:intent_status}]
status:
  phase: Applied
  fulfillmentState: FULFILLED
  observedGeneration: 1
  intentGroupStatuses:
    - id: embb
      domainStatus:
        coreDomain:
          state: CONFIGURED
          message: "UEs registered with 5QI=6"
        ranDomain:
          state: CONFIGURED
          message: "Slice configured: SST=1, SD=66051"
      achievedTargets:
        bandwidth: achieved
        latency: not_applicable
\end{lstlisting}

\subsubsection{基於 Reconciliation 的自癒機制}
此狀態回饋機制直接與編排器的 \textit{Reconciler} 連動。當 \texttt{status} 顯示意圖處於 \texttt{NOT\_FULFILLED} 或發生配置漂移（Configuration Drift）時，控制迴圈會自動再次觸發南向派發流程。例如，若核心網用戶註冊因網路抖動失敗，編排器將在下一次調和週期中自動重試 \texttt{RegisterSubscriber} 操作，直至狀態達到「最終一致性」。這種基於 CRD Status 的設計，使系統具備了基礎的配置自癒（Self-healing）能力。

\section{srsRAN 領域編排器與雲原生調和實作 (srsRAN Operator / RAN NSSMF)}
\label{sec:ran_nssmf_implementation}

本研究開發之 \texttt{srsran-operator} 擔任 3GPP 架構中的存取網切片子網域管理功能（RAN NSSMF）。該組件採用雲原生 Operator 模式，實作於工作負載叢集（Workload Cluster）中，負責將抽象的網路功能部署宣告轉化為實體的 \textit{srsRAN} 網元配置。本節將詳細論述其調和工作流與跨叢集生命週期管理機制。

\subsection{NFDeployment 調和工作流 (Reconciliation Workflow)}
\label{subsec:nfdeployment_workflow}

\texttt{srsran-operator} 的核心邏輯建立在對 Nephio \texttt{NFDeployment} 自定義資源（CR）的持續監聽與調和上。為了符合電信網路配置的多維度需求，本系統實作了一套「引用的遞迴查找（Reference Walking）」機制，確保配置的階層化與模組化。

\subsubsection{引用遞迴查找機制 (Reference Walking)}
Operator 並不直接在單一資源中讀取所有參數，而是透過 KRM 的引用規範進行深度查找：

\begin{enumerate}
    \item \textbf{入口點識別}：Operator 監聽 \texttt{NFDeployment} 資源。當偵測到 \texttt{spec.provider} 標註為 \texttt{gnb.srsran.io} 時，觸發調和程序。
    \item \textbf{第一層查找 (parametersRefs)}：Reconciler 首先解析 \texttt{spec.parametersRefs} 欄位，尋找關聯的 \texttt{NFConfig} 資源。此步驟定義了該部署實例的基本框架。
    \item \textbf{第二層查找 (configRefs)}：進入 \texttt{NFConfig} 後，Operator 進一步遍歷其 \texttt{spec.configRefs} 列表。此時，系統會執行遞迴查找，分別提取以下三個領域特定的配置資源：
    \begin{itemize}
        \item \textbf{\texttt{SrsRANConfig}}：定義容器鏡像版本、AMF 對接位址與 ZMQ 模擬參數。
        \item \textbf{\texttt{SrsRANCellConfig}}：定義無線單元（Cell）物理參數與轉譯後的切片調度權重（PRB Ratio/Priority）。
        \item \textbf{\texttt{PLMNConfig}}：定義行動國家代碼與支援的切片清單（SST/SD）。
    \end{itemize}
    \item \textbf{資源聚合與依賴檢查}：唯有當上述所有關聯資源皆成功讀取且校驗通過後，Operator 才會進入下一階段的實體資源生成，否則將於 \texttt{Status} 回報 \texttt{invalidConfigInfo} 錯誤並進入指數退避重試。
\end{enumerate}

\subsubsection{跨叢集生命週期管理與 GitOps 整合}
本系統的佈署流程體現了 \textbf{GitOps} 與 \textbf{KRM} 的深度整合。資源的生命週期並非由傳統指令觸發，而是透過宣告式管線進行流轉：

\begin{itemize}
    \item \textbf{管理叢集編排 (Management Plane)}：編排器（NSMF）透過 \textit{Nephio Porch} 操控藍圖套件（Blueprint Package）。利用 \texttt{PackageVariantSet} 機制，將針對特定工作負載叢集客製化（Specialization）後的套件發佈至對應的 Git 儲存庫。
    \item \textbf{單一事實來源 (Git as SSOT)}：Git 儲存庫作為存取網配置的單一事實來源。所有的意圖變更（如增加切片）最終皆會反應為 Git 上的 YAML 修訂版本。
    \item \textbf{狀態同步 (ConfigSync)}：部署於工作負載叢集的 \textit{ConfigSync} 代理程式會持續拉取 Git 變更，並將 \texttt{NFDeployment} 與相關的 \texttt{Config} 資源套用（Apply）至叢集中。
    \item \textbf{本地實體化 (Local Execution)}：一旦 \texttt{NFDeployment} 資源出現在工作負載叢集內，\texttt{srsran-operator} 隨即捕捉到此訊號，並執行前述之遞迴查找，最終生成 CU-CP、CU-UP 與 DU 的 \texttt{Deployment} 與 \texttt{ConfigMap}，完成從雲端編排到邊緣實施的自動化閉環。
\end{itemize}

透過此機制，本系統不僅達成了切片部署的自動化，更確保了在多叢集電信雲環境下，配置的一致性與可追溯性。

\subsection{階層化配置模型與自定義資源 (Hierarchical Config CRDs)}
\label{subsec:hierarchical_config_model}

為了在雲原生環境中實現電信級網元的靈活配置，本研究針對 \textit{srsRAN} 網元的特性，設計了一套階層化的配置模型。此模型將原本雜亂的單一組態檔解構為三個維度的自定義資源（Custom Resources），分別對應電信網路中不同層級的參數需求，從而支持編排器執行精準的「套件專化（Package Specialization）」。

\subsubsection{SrsRANConfig：應用與模擬環境層 (Application Layer)}
\texttt{SrsRANConfig} 資源主要定義與容器運行環境及模擬基礎設施相關的參數。其核心功能在於解耦網元邏輯與底層部署環境：
\begin{itemize}
    \item \textbf{容器鏡像管理}：定義 CU-CP、CU-UP 與 DU 各組件所使用的容器鏡像版本。
    \item \textbf{ZMQ 虛擬射頻配置}：定義是否啟用 ZMQ 模擬模式，並設定採樣率（\texttt{srate}）、發射增益（\texttt{txGain}）與接收增益（\texttt{rxGain}），確保 SDR 模擬環境的穩定性。
    \item \textbf{核心網對接資訊}：定義目標 AMF 的存取位址，作為 CU-CP 建立 N2 連結的基礎。
\end{itemize}

\subsubsection{SrsRANCellConfig：物理與資源調度層 (Physical \& Scheduling Layer)}
此資源是落實存取網切片意圖的核心關鍵，主要定義 Distributed Unit (DU) 的實體單元參數以及 \textbf{RAN Slicing Scheduler Policy}：
\begin{itemize}
    \item \textbf{無線單元參數}：包含下行絕對頻點識別碼（\texttt{dlArfcn}）、運作頻段（\texttt{band}）、通道頻寬與子載波間隔（\texttt{commonScs}）等物理層配置。
    \item \textbf{切片排程策略}：此欄位直接承載來自 \textit{E2E Orchestrator} 轉譯後的意圖參數。透過 \texttt{spec.slicing} 陣列，為每個 S-NSSAI 定義專屬的 \texttt{minPrbPolicyRatio}、\texttt{maxPrbPolicyRatio} 與排程優先級（\texttt{Priority}），實現無線資源的硬性隔離與差別化服務。
\end{itemize}

\subsubsection{PLMNConfig：網路識別層 (Network Identity Layer)}
\texttt{PLMNConfig} 資源定義了行動網路的識別邏輯，確保用戶設備（UE）能正確識別並接取目標切片。其配置內容主要影響 CU-CP 廣播的系統資訊：
\begin{itemize}
    \item \textbf{網路標識}：定義公共陸地行動網路標識（PLMN ID）與追蹤區碼（Tracking Area Code, TAC）。
    \item \textbf{切片支援列表}：定義該追蹤區內支援的切片清單（S-NSSAI List）。編排器會動態同步此處的 SST/SD 資訊，使其與 \texttt{SrsRANCellConfig} 中的排程器配置保持一致，避免因識別碼不匹配導致的 UE 接取失敗。
\end{itemize}

透過此階層化模型，\texttt{srsran-operator} 能夠將來自不同維度的宣告式狀態，最終聚合（Aggregate）並渲染為網元所需的實體組態檔，達成高度自動化的資源調和。

\subsection{電信雲化網路拓樸與邏輯介面管理 (Multus \& IPAM)}
\label{subsec:multus_ipam_management}

為了在虛擬化環境中精確模擬 5G 實體網元的連通行為，本研究定義了一套「確定性的邏輯網路拓樸（Deterministic Logical Topology）」。透過 \textit{Nephio IPAM} 與 \textit{Multus CNI} 的協作，系統得以自動將預定義的 IP 地址注入對應的虛擬介面中。這確保了在軟體定義無線電（SDR）模擬環境下，CU、DU 與 UE 之間的 ZMQ 通道能具備穩定的路由路徑。

\subsubsection{多介面容器設計與邏輯隔離}
在 3GPP 標準架構中，控制面（Control Plane）與用戶面（User Plane）的物理或邏輯隔離是確保電信網路安全與效能的關鍵。傳統 Kubernetes Pod 通常僅具備單一網路介面（\texttt{eth0}），難以模擬 5G 網元複雜的介面需求（如 N2 控制面與 N3 用戶面的分離）。

本系統透過 \textit{Multus CNI} 為 \textit{srsRAN} 組件實作了多介面容器設計。藉由 \textit{NetworkAttachmentDefinition} 資源，每個網元 Pod 能夠掛載多個專屬的次要介面（Secondary Interfaces），分別承載 N2、N3、E1 與 F1 等不同性質的流量，達成邏輯上的徹底隔離。

\subsection{無線資源隔離與切片調度轉譯 (RAN Slicing Realization)}
\label{subsec:ran_slicing_realization}

在 \textit{E2E Orchestrator} 完成跨網域意圖轉譯後，\texttt{srsran-operator} 需將接收到的領域參數轉化為 \textit{srsRAN} 分散單元（DU）可執行的實施配置。

\subsubsection{從宣告式狀態至實體組態之渲染}
\texttt{srsran-operator} 內建了基於 Go Template 實作的組態渲染引擎。當調和工作流讀取到 \texttt{SrsRANCellConfig} 中的 \texttt{spec.slicing} 內容時，會將其轉換為 \textit{srsRAN} 專屬的 YAML 結構。

\subsubsection{切片調度參數映射策略}

\begin{itemize}

    \item \textbf{調度優先權保障}：透過賦予 URLLC 高達 200 的排程優先級，\textit{srsRAN} 的 MAC 層排程器會在每個時槽（Slot）優先處理該切片的緩存數據，從而縮短排隊延遲（Queuing Delay），落實意圖中對「超低延遲」的保障。

\subsection{資源狀態追蹤與自癒回饋機制 (Status Reporting \& Self-healing)}
\label{subsec:ran_status_reporting}

為了與上層 \textit{E2E Orchestrator} 達成閉環連動，\texttt{srsran-operator} 實作了細粒度的資源狀態回報與異常處理機制。不同於編排器關注的業務意圖達成率，本節著重於 Operator 如何管理底層實體資源的生命週期完整性。

\subsubsection{基於 NFDeployment Conditions 的閉環監測}
\texttt{srsran-operator} 透過動態更新 \texttt{NFDeployment} 資源的 \texttt{status.conditions} 欄位，將工作負載叢集內的實體佈署狀態回饋至 Nephio 管理層。
當 Operator 成功解析所有依賴之 CR 資源並渲染出 \textit{srsRAN} 所需之 \texttt{Deployment} 與 \texttt{ConfigMap} 時，系統會將 \texttt{resourceCreation} 指標標註為 \texttt{True}，並隨後將整體狀態設為 \texttt{Ready}。

此機制確保了狀態的透明度，使得上層編排器能精確判別存取網域的參數是否已從「宣告狀態」轉化為「執行狀態」。若底層資源發生異常（如 Pod 啟動失敗），Operator 會即時更新狀態訊息，觸發整體的狀態回饋迴圈。

\subsubsection{錯誤處理：配置缺失之回饋機制}
在調和流程中，最常見的異常為依賴資源的缺失。本系統實作了 \texttt{invalidConfigInfo} 檢測邏輯：
當 Operator 執行引用遞迴查找時，若發現 \texttt{SrsRANConfig}、\texttt{SrsRANCellConfig} 或 \texttt{PLMNConfig} 其中任一項資源尚未就緒，Reconciler 會在 \texttt{NFDeployment.status.conditions} 中注入類型為 \texttt{invalidConfigInfo} 的失敗訊息，進入重試迴圈，直到所有依賴資源皆齊備為止

\subsubsection{資源清理之原子性保障：Finalizer 機制}
為確保網路功能的生命週期管理符合「原子性（Atomicity）」，本研究在 Operator 中實作了 \texttt{Finalizer} 刪除鎖定機制。
傳統的 Kubernetes 刪除操作若未妥善處理，可能會導致 \texttt{NFDeployment} 資源已刪除，但其派生出的 CU/DU \texttt{Deployment} 仍殘留在叢集中的「資源遺留（Resource Leak）」現象。

當用戶或編排器發起刪除請求時，\texttt{srsran-operator} 會攔截該事件並執行清理邏輯，確保與該 \texttt{NFDeployment} 關聯的所有 \texttt{ConfigMap}、\texttt{ServiceAccount} 與 \texttt{Deployment} 皆被同步銷毀。唯有在確認所有底層資源皆已安全釋放後，Operator 才會移除 \texttt{Finalizer} 並允許該網元實例從叢集中徹底消失，從而保障了網路切片資源清理的完整性。

\section{核心網網域之自動化編排與特殊化實作}
\label{sec:core_domain_orchestration}

本章節論述本系統如何針對核心網網域實作自動化編排。不同於存取網端的高度自定義實作，核心網網域側重於利用 \textit{Nephio} 標準架構進行異質網元的配置專化（Specialization），並透過自動化管道克服現有開源藍圖的實施挑戰。

\subsection{核心網網域編排器概覽與特殊化策略}
\label{subsec:core_orchestrator_overview}

\subsubsection{組件定位與架構相似性}
在本系統中，核心網端的資源調和由 \texttt{free5gc-operator} 負責。該組件採用 \textit{Nephio} 官方發佈版本，其核心運作邏輯同樣基於 Kubernetes Operator 模式。由於 \texttt{free5gc-operator} 在監聽 \texttt{NFDeployment} 資源、執行調和（Reconcile）迴圈以及透過 KRM 狀態機維護資源最終一致性等方面，與前述 \ref{sec:ran_nssmf_implementation} 節所論述之 \texttt{srsran-operator} 具備高度架構相似性，故本節不再重複贅述其基礎組件之底層運作原理，而將重點轉向其上層編排策略。

\subsubsection{實作挑戰與藍圖瑕疵說明}
在實作過程中發現，現有的 \textit{Nephio} 官方核心網範本（Blueprints）在特定實驗環境下存在配置瑕疵。這些挑戰主要包含：
\begin{itemize}
    \item \textbf{IP 地址衝突}：官方藍圖在處理跨叢集 IPAM 分配時，若缺乏精確的特殊化干預，容易產生介面位址重疊之隱患。
    \item \textbf{資源池數量不足}：預設配置通常僅針對單一網路切片進行定義，無法滿足本研究中 eMBB 與 URLLC 等多維度多切片（Multi-slice）共存的資源池隔離需求。
\end{itemize}
因此，本系統必須在編排流程中引入額外的介入機制，以修正並強化藍圖的健全性。

\subsubsection{自動化編排管道 (Domain Orchestration Pipeline)}
為了達成核心網網元的自動化部署並確保其能與 \textit{NSMF}（即 \textit{E2E Orchestrator}）有效對接，本研究設計了一套自動化編排管道。此管道透過自動化腳本實作了 \textit{NSSMF} 層級的「特殊化（Specialization）」流程：
\begin{enumerate}
    \item \textbf{配置注入}：將轉譯後的特定切片參數動態注入套件中。
    \item \textbf{依賴協同}：自動協調 SMF 與 UPF 間的端點依賴關係。
    \item \textbf{狀態觸發}：驅動 \textit{Porch} 的生命週期管理，使配置變更生效。
\end{enumerate}
本節後續將詳細展開此管道在基礎設施配置與切片感知特殊化方面的具體實作。

\subsection{雲原生基礎設施自動化：電信拓樸與虛擬介面配置}
\label{subsec:infra_automation}

核心網網元的自動化編排不僅限於應用層的佈署，更包含對底層網路環境的預先整備。本系統透過自動化腳本確保工作負載叢集（Workload Cluster）之節點具備承載 5G 邏輯介面的物理基礎，並向管理平台宣告精確的網路視圖。

\subsubsection{基礎設施即代碼 (IaC)：虛擬介面與 VLAN 隔離實作}
本研究採用基礎設施即代碼（Infrastructure as Code, IaC）之原則，透過自動化指令集在 Workload Cluster 的 Worker 節點中建立必要的網路底層。具體實作包含：
\begin{itemize}
    \item \textbf{eth1 虛擬介面建立}：利用 Linux \texttt{dummy} 介面技術建立名為 \texttt{eth1} 的虛擬主幹介面，作為電信流量的數據平面（Data Plane）入口。
    \item \textbf{多維度 VLAN 劃分}：在 \texttt{eth1} 之上建立 VLAN 2 至 6 的子介面（如 \texttt{eth1.3} 承載 N2/N3 流量，\texttt{eth1.4} 承載 N4 流量）。這種封裝技術在單一物理網卡上實作了邏輯上的完全隔離，符合 3GPP 對於不同參考點（Reference Points）安全性與效能隔離的要求。
\end{itemize}

\subsubsection{宣告式網路拓樸定義 (RawTopology)}
為了讓 \textit{Nephio} 管理叢集能識別並管理工作負載叢集的網路資源，本系統下發了名為 \texttt{RawTopology} 的宣告式資源。該資源定義了叢集節點的物理屬性與標籤（Labels），其學術與實務價值在於：
\begin{itemize}
    \item \textbf{建立物理連通視圖}：透過 \texttt{RawTopology}，管理平面得以獲知工作負載叢集的節點名稱與其所屬的提供者資訊，建立起管理叢集與受控叢集間的邏輯映射。
    \item \textbf{IP 分配與路由之邏輯基礎}：此定義為後續 \textit{Nephio IPAM} 的自動化分配提供了上下文。系統能根據拓樸定義中的叢集名稱標籤（\texttt{nephio.org/cluster-name}），自動在對應的 \texttt{NetworkInstance} 中劃分出專屬的 IP 地址池與路由條目，確保核心網網元在啟動時能獲得具備唯一性且可路由的電信級 IP 地址。
\end{itemize}

透過此自動化整備流程，系統消除了手動配置網路介面的繁瑣與出錯風險，為後續網元套件的特殊化流程奠定了堅實的基礎。

\subsection{切片感知之 Package 特殊化與多維度資源池配置}
\label{subsec:slice_aware_specialization}

核心網編排的挑戰在於如何將通用的網元藍圖轉換為支援多切片架構的實體配置。本系統透過自動化特殊化程序，針對 UPF 的資源池與 SMF 的關聯配置進行動態調整。

\subsubsection{套件變更 (Mutate) 機制：多維度資源池擴展}
在預設的核心網藍圖中，用戶面網元（UPF）通常僅配置單一的網路位址池，難以實作異質業務間的流量隔離。本系統的自動化管道在 \textit{NSSMF} 層級實作了套件變更機制：
\begin{itemize}
    \item \textbf{動態修改 UPF \texttt{NFDeployment}}：管道透過 \textit{Porch} 提取 UPF 套件草稿，並針對 \texttt{upfdeployment.yaml} 進行注入。
    \item \textbf{資源池三重化}：將預設的單一 \texttt{vpc-internet} 資源池擴展為三組獨立的子網段（如 10.0.1.0/24、10.0.2.0/24 與 10.0.3.0/24），分別對應 eMBB、URLLC 與 MIoT 三種不同切片類型。這種配置確保了不同業務類型的用戶設備（UE）在核心網端具備獨立的 IP 路由路徑，從源頭落實數據平面的切片隔離。
\end{itemize}

\subsubsection{跨組件依賴協同：SMF 與 UPF 之配置對映}
核心網控制面（SMF）必須精確掌握用戶面（UPF）的端點資訊才能正確執行會話管理。為了達成網域內部的服務鏈結（Service Chaining），本系統實作了跨組件的依賴調和：
\begin{itemize}
    \item \textbf{宣告式引用 (Step 7)}：系統建立了一個 \texttt{ref.nephio.org} 規範的 \texttt{Config} 資源，作為 SMF 與 UPF 間的資訊橋樑。
    \item \textbf{端點自動對映}：該 \texttt{Config} 資源會精確引用 UPF 生成的 N3（RAN 介面）、N4（控制面介面）與 N6（外部數據介面）之端點位址。透過此機制，SMF 的 Operator 能自動獲取 UPF 的實體配置，確保會話建立請求能準確導向對應切片的資源池，達成網域內部組件間的無縫協同。
\end{itemize}

透過此特殊化與依賴協同機制，本系統成功將原本靜態的核心網藍圖，轉化為具備切片感知能力的動態服務架構。

\section{實驗驗證環境與多用戶模擬架構}
\subsection{驗證環境之編排策略與職能分工}
\label{subsec:validation_strategy}

為了驗證意圖驅動編排系統在真實電信場景下的有效性，本研究構建了一個模擬實驗平台。在編排策略上，本系統採取了「基礎設施與測試資源分離」的設計原則，將受測系統（System Under Test, SUT）與模擬器（Emulator）在編排層級進行邏輯解耦。

\subsubsection{基礎設施與測試資源的編排差異}
在本實驗環境中，網路元件依其職能被劃分為兩個不同的編排管道：
\begin{itemize}
    \item \textbf{Nephio 自動化管道 (受測系統)}：核心網元件（free5GC）與存取網組件（srsRAN CU/DU）被視為電信基礎設施的一部分。這些組件被納入基於 \textit{Nephio} 的 \textit{workload pipeline} 中，接受 \textit{E2E Orchestrator} 的宣告式意圖管理。這模擬了電信營運商在生產環境中對網路功能（NFs）的自動化編排與生命週期管理。
    \item \textbf{Helm 部署層 (測試與模擬資源)}：用戶設備（srsUE）、\textit{GNU Radio Breaker}（射頻橋接器）以及 \textit{iperf3} 流量產生工具，則被歸類為測試端資源。本研究利用 \texttt{srsran-helm} 專案，透過手動或腳本化的 \textit{Helm} 方式進行部署。UE 等元件無需納入複雜的 \textit{Nephio} 宣告式管線，而是以更為直接、靈活的方式建立。
\end{itemize}

\subsubsection{邏輯解耦與 SUT/Emulator 分離原則}
此種職能分工的核心價值在於實現了「模擬器（Emulator）」與「受測系統（SUT）」的邏輯解耦，符合電信一致性測試之標準原則。具體優勢如下：
\begin{enumerate}
    \item \textbf{動態擴展能力}：測試人員可以在不變動、不重啟底層核心網與存取網基礎設施的前提下，透過 \textit{Helm} 快速增減測試用戶（UE）的數量或調整 \textit{GNU Radio} 的通道模型。這種設計允許系統在穩定的實體網路拓樸上，執行多樣化的邊界條件測試。
    \item \textbf{環境真實性模擬}：將 CU/DU 置於 \textit{Nephio} 監管下，能真實反映意圖轉譯與配置下發的時耗與準確性；而將 UE 獨立於編排管線之外，則能避免測試流量干擾編排面的信令交換，從而獲取更純粹的服務效能數據（如 Latency 與 PRB 佔用率）。
    \item \textbf{測試腳本化與可重複性}：透過 \textit{Makefile} 封裝的 \textit{Helm} 指令（如 \texttt{make ue1}、\texttt{make gnu}），實驗環境能快速恢復至初始狀態，確保了跨切片驗證實驗的可重複性與嚴謹性。
\end{enumerate}

透過此編排策略，本研究建立了一個既能體現雲原生自動化編排特性，又具備高度測試靈活性的實驗床（Experimental Bed），為後續的端到端切片隔離驗證提供了堅實的架構基礎。

\subsection{基於 GNU Radio 之多用戶射頻橋接機制}
\label{subsec:gnuradio_bridge}

在端到端切片驗證中，模擬多用戶（Multi-UE）併發存取的情境是評估資源隔離效能的關鍵。然而，在雲原生模擬環境中實作此架構面臨著特定的技術限制，本節論述如何透過 \textit{GNU Radio} 組件克服這些障礙並實作虛擬射頻橋接。

\subsubsection{ZMQ 模擬限制之克服}
\textit{srsRAN} 所提供的 ZMQ 模擬驅動程式在預設情況下主要設計為一對一的點對點連結。在標準配置中，分散單元（DU）與用戶設備（UE）之間的射頻訊號透過 TCP Socket 直接進行串流交換。

這種點對點架構無法直接模擬行動通信中的廣播（Broadcast）與多路存取（Multiple Access）特性。若要驗證異質切片（如 eMBB 與 URLLC）在同一小區（Cell）下的競爭與隔離，系統必須具備將單一 DU 訊號分發至多個虛擬 UE 的能力。為此，本研究引入了 \textbf{GNU Radio Breaker} 作為射頻中繼層，打破了原本 1:1 的連線限制，建立起 1:N 的虛擬無線通道。

\subsubsection{虛擬通道邏輯與流圖實作}
本系統透過 \textit{GNU Radio Companion} 建立了一個名為 \texttt{multi\_ue\_scenario.grc} 的流圖（Flow Graph），並將其編譯為 Python 腳本運行於獨立容器中。該組件在邏輯上扮演了實體空間中的無線電傳播媒介，其核心邏輯如下：

\begin{enumerate}
    \item \textbf{下行鏈路（Downlink）分發}：橋接器監聽來自 DU 的 ZMQ 端點，接收下行 IQ 數據流後，利用 \texttt{Broadcast} 邏輯將相同的訊號流同步複製並轉發至多個 UE 的接收端點。
    \item \textbf{上行鏈路（Uplink）合併}：橋接器同時監聽來自所有在線 UE 的上行 ZMQ 端點。當多個 UE 同時發送訊號時，橋接器會在 IQ 層級執行加法運算（Summation）進行訊號合併，最後將混合後的訊號流回傳至 DU 的接收端，模擬實體天線接收到的多用戶疊加訊號。
\end{enumerate}

\subsubsection{虛擬射頻鏈路之端點對映}
為了確保各組件在分散式環境下能精確連通，本研究定義了一套確定性的 ZMQ 端點與埠號對映規則。表 \ref{tab:zmq_endpoint_mapping_detail} 詳細列出了 DU、橋接器與各個 UE 之間的邏輯鏈路對應關係。

\subsection{多切片用戶設備（Multi-Slice UE）配置實作}
\label{subsec:multi_slice_ue_config}

在本研究的驗證體系中，用戶設備（UE）端點必須能精確模擬異質業務的接入行為。透過 \texttt{srsran-helm} 專案，本系統實作了具備切片感知能力的多用戶配置架構，確保每個 UE 實例能與核心網及存取網中預定義的特定切片正確對接。

\subsubsection{參數映射與 IMSI 管理}
為了在模擬環境中區分不同業務屬性的用戶，本研究利用 Helm 的 \texttt{values.yaml} 機制實作了靜態參數映射。每個 UE 實例（UE1 與 UE2）均分配有唯一的國際行動用戶識別碼（IMSI），該識別碼與核心網 \textit{free5GC} 中的用戶訂閱資料嚴格對應。

在配置邏輯上，UE1 被預定義為 eMBB 業務終端，其配置檔中指定了對應的 S-NSSAI（SST: 1, SD: 0x010203）；而 UE2 則被預定義為 URLLC 業務終端，請求 S-NSSAI（SST: 1, SD: 0x112233）。透過在 Helm 模板中將這些識別碼與特定的 ZMQ 虛擬介面埠號進行綁定，系統能在啟動時自動為不同切片建立專屬的射頻連線路徑。

\subsection{端到端服務性能驗證工具鏈}
\label{subsec:validation_toolchain}

為量化驗證意圖驅動編排系統對網路切片之性能保障能力，本研究建立了一套標準化的性能驗證工具鏈。該工具鏈旨在自動化產生測試負載，並確保實驗數據在不同意圖配置下具備可重複性與嚴謹性。

\subsubsection{流量產生機制與 Namespace 穿透技術}
性能驗證之核心在於模擬真實業務負載。本系統利用 \texttt{iperf3} 工具產生受控的 UDP 流量，以模擬 eMBB 的大頻寬需求與 URLLC 的持續性小封包傳輸。

由於 \textit{srsUE} 運行於獨立的容器環境中，其產生的切片流量經由核心網分發後，會導向用戶設備內的專屬虛擬網卡（\texttt{uesimtun0}）。為測得精確之端到端數據，測試程序需透過 Linux 的 \texttt{nsenter} 技術，穿透容器邊界並進入該 Pod 專屬的網路命名空間（Network Namespace）。透過在此命名空間內指定 \texttt{uesimtun0} 作為 I/O 介面，流量得以直接經由 5G 協議棧進行傳輸，從而排除宿主機網路環境的干擾。

\subsubsection{自動化測試工作流 (Makefile Workflow)}
為了提升實驗效率並消除人為操作之不確定性，本研究將複雜之部署、配置與壓測指令封裝於 \texttt{Makefile} 中，構建了自動化測試工作流：
\begin{itemize}
    \item \textbf{資源快速佈署}：透過 \texttt{make ue1} 與 \texttt{make ue2} 指令，系統能依照預定義的切片映射表（IMSI 與 S-NSSAI）快速啟動測試終端。
    \item \textbf{可重複實驗路徑}：利用 \texttt{make iperf-ue1} 等封裝後的測試指令，系統會自動定位容器進程、進入命名空間並執行帶有特定參數（如頻寬、持續時間）的 \texttt{iperf3} 客戶端。
\end{itemize}

此套工具鏈將原本繁瑣的環境準備與數據採集過程轉化為可程式化的步驟，確保了第五章性能評估中，針對不同資源隔離策略（PRB Ratio 與 Priority）之測試結果皆能於一致且客觀的基準線上進行對比。
